# Quiz Parser ‚Äî ANTLR4 Project

A parser for a custom `.qm` file format, built with [ANTLR4](https://www.antlr.org/) and Python.

---

## How It Works ‚Äî Pipeline Overview

```
quiz file (.qm)
      ‚îÇ
      ‚ñº
  QuizLexer          tokenises raw text into a stream of tokens
      ‚îÇ
      ‚ñº
  QuizParser         builds a parse tree according to Quiz.g4 grammar rules
      ‚îÇ
      ‚ñº
  QuizVisitorImpl    walks the parse tree, returns typed Python dataclass objects
      ‚îÇ
      ‚ñº
  print_quiz()       renders the result to the terminal
```

---

## File Structure

| File | Role |
|---|---|
| `Quiz.g4` | ANTLR grammar ‚Äî defines the language |
| `QuizLexer.py` | **Generated** ‚Äî breaks input text into tokens |
| `QuizParser.py` | **Generated** ‚Äî parses token stream into a parse tree |
| `QuizListener.py` | **Generated** ‚Äî base listener interface (enter/exit hooks per rule) |
| `QuizVisitor.py` | **Generated** ‚Äî base visitor interface (one `visit*` method per rule) |
| `QuizVisitorImpl.py` | **Written** ‚Äî concrete visitor, extracts data from the parse tree |
| `models.py` | **Written** ‚Äî plain Python dataclasses representing the parsed data |
| `main.py` | **Written** ‚Äî entry point, wires everything together |

The four `Quiz*.py` files are auto-generated by ANTLR and should **not** be edited by hand. Regenerate them with:

```bash
antlr4 -Dlanguage=Python3 -visitor Quiz.g4
```

## The Grammar (`Quiz.g4`)

The grammar describes what a valid `.qm` file looks like. It has two sections:

### Parser Rules (structure)

```
quiz          ‚Üí  'QUIZ:' title themeBlock? metadata* question+
themeBlock    ‚Üí  'THEME' '{' themeStatement+ '}'
themeStatement‚Üí  key '=' value
               | selector '[' attr=val, ... ']'
selector      ‚Üí  'quiz' | 'question' | 'answer' | 'correct' | 'wrong'
metadata      ‚Üí  IDENTIFIER ':' (text | NUMBER | BOOLEAN)
question      ‚Üí  'QUESTION' points? ':' text image? answer+
points        ‚Üí  '(' 'points=' NUMBER ')'
answer        ‚Üí  OPTION_LABEL ':' answerContent '*'?
answerContent ‚Üí  text image? | image
image         ‚Üí  'IMAGE:' (STRING_LITERAL | text)
```

### Lexer Rules (tokens)

| Token | Pattern | Example |
|---|---|---|
| `OPTION_LABEL` | `A`‚Äì`E` | `A`, `B`, `C` |
| `CORRECT_MARKER` | `*` | marks the correct answer |
| `BOOLEAN` | `true` / `false` | metadata values |
| `IDENTIFIER` | letter then `[a-zA-Z0-9_]*` | `TIME_LIMIT`, `font` |
| `NUMBER` | digits, optional decimal and `%` | `60`, `5`, `14` |
| `STRING_LITERAL` | `"..."` | `"assets/eiffel.jpg"` |
| `TEXT_CONTENT` | alphanumeric + spaces/punctuation | `What is 2 plus 2?` |
| `WS` | whitespace | skipped |

> **Note:** Because `TEXT_CONTENT` greedily consumes spaces, some formatting rules apply to `.qm` files ‚Äî see the [Input Format](#input-format) section below.

## The Visitor (`QuizVisitorImpl.py`)

`QuizVisitorImpl` extends the generated `QuizVisitor` base class and overrides every `visit*` method. Each method maps 1-to-1 with a grammar rule and returns a typed value:

| Method | Grammar rule | Returns |
|---|---|---|
| `visitQuiz` | `quiz` | `QuizData` |
| `visitThemeBlock` | `themeBlock` | `ThemeData` |
| `visitThemeStatement` | `themeStatement` | mutates `ThemeData` |
| `visitSelector` | `selector` | `str` |
| `visitAttributeList` | `attributeList` | `dict` |
| `visitAttribute` | `attribute` | `(key, value)` tuple |
| `visitMetadata` | `metadata` | `(key, value)` tuple |
| `visitQuestion` | `question` | `QuestionData` |
| `visitPoints` | `points` | `int` or `float` |
| `visitQuestionMedia` | `questionMedia` | `str` (image path) |
| `visitAnswer` | `answer` | `AnswerData` |
| `visitAnswerContent` | `answerContent` | `(text, image)` tuple |
| `visitImage` | `image` | `str` (path, quotes stripped) |
| `visitValue` | `value` | `str`, `int/float`, or `bool` |
| `visitText` | `text` | `str` |

This differs from the alternative **Listener** pattern (also generated). A listener uses event callbacks (`enterQuiz`, `exitQuiz`, ‚Ä¶) fired automatically as ANTLR walks the tree ‚Äî useful for side-effects like building a symbol table. The visitor pattern used here is better suited for building a return value (the data model) because each `visit*` call explicitly returns data up the call stack.

## Entry Point (`main.py`)

`main.py` only does three things:

1. **Lex + Parse** ‚Äî feeds the file through `QuizLexer` and `QuizParser`
2. **Visit** ‚Äî calls `QuizVisitorImpl().visitQuiz(tree)` to get a `QuizData` object
3. **Print** ‚Äî passes the `QuizData` to `print_quiz()` for display

## Input Format

A `.qm` file must follow these conventions:

- **Quiz title** ‚Äî any text/identifier after `QUIZ:`
- **Theme block** (optional) ‚Äî `THEME` keyword, then `{` on the same or next line; no space between a selector keyword and `[`
- **Metadata** (optional, repeatable) ‚Äî `KEY: value` lines between the theme block and first question
- **Questions** ‚Äî `QUESTION:` or `QUESTION(points=N):` (no space before `(`)
- **Answers** ‚Äî `A:` through `E:`, with `*` after the content to mark correct
- **Images** ‚Äî `IMAGE: "path/to/file"` quoted paths, either on the question line or as the answer content
- **Attribute values** inside `[...]` must be quoted strings (`"Verdana"`) or bare numbers

### Example

```
QUIZ: Science 101
THEME
{
    primary_color = "blue"
    question[font="Verdana", size=14]
    correct[color="green", weight="bold"]
}
TIME_LIMIT: 60
DIFFICULTY: Easy
CATEGORY: Biology

QUESTION: Which of these is a mammal?
A: Shark
B: Whale *
C: Lizard
```

### Example with image

```
QUIZ: World Landmarks
QUESTION(points=5): Name this structure
IMAGE: "assets/eiffel.jpg"
A: Eiffel Tower *
B: Statue of Liberty
C: Colosseum
```
